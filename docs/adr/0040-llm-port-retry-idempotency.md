# ADR 0040 — LLM port: retry, timeout, idempotency

- Status: Accepted
- Date: 2026-05-16
- Deciders: CTO
- Drives: [SIN-62901](/SIN/issues/SIN-62901) (this ADR), [SIN-62196](/SIN/issues/SIN-62196) (Fase 3 parent)
- Cross-links: [ADR 0041](./0041-anonymizer-pii.md) (anonymizer is the first port middleware), [ADR 0043](./0043-token-debito-atomico.md) (wallet reserve/commit straddles the LLM call)

## Context

Fase 3 ([SIN-62196](/SIN/issues/SIN-62196)) introduces the
operator-facing AI assist surface: the CRM calls a language model on
the operator's behalf to summarise a conversation, draft a reply, or
extract intent. The model lives behind a paid API (OpenRouter, with
other providers planned). Every call costs tokens, blocks an operator
on a UI, and can fail in three independent ways:

- **Transient transport failure** — TCP reset, TLS handshake error,
  upstream 502 from OpenRouter's edge. The same request retried 250ms
  later usually succeeds.
- **Provider-side rate limit / capacity** — OpenRouter returns 429 or
  503. Backing off and retrying within seconds is the correct
  response.
- **Provider-side hard error** — malformed prompt, content-policy
  block, model removed. Retrying does not help; the caller needs to
  see the error.

The CRM also has a stricter constraint than most LLM consumers: the
operator panel issues the same logical request twice when the
operator clicks the button twice, when the page reloads mid-call, or
when our retry path overlaps with a slow provider response.
Double-billing the tenant for the same logical request is
unacceptable — wallets debit on commit ([ADR
0088](./0088-wallet-basic.md), [ADR 0043](./0043-token-debito-atomico.md)),
and operators reading the same answer twice is a usability bug.

The naive options are tempting and wrong:

- **Option A — `net/http` call directly from the use-case.** The use-
  case imports the vendor SDK, hand-rolls retries with `time.Sleep`,
  and inlines OpenRouter-specific error parsing. Pollutes the AI
  domain core with transport concerns and breaks the hexagonal
  boundary already enforced for `internal/ai/port` ([ADR
  0077](./0077-llm-prompt-isolation.md)-style lint rules apply).
- **Option B — Idempotency at the wallet only.** The wallet has a
  `(tenant_id, conversation_id, request_id)` key already (see [ADR
  0043](./0043-token-debito-atomico.md) §D2). Rely on the wallet to
  short-circuit duplicates and let the LLM see them all. Rejected
  because the wallet shortcut happens *after* the LLM has already
  been called and billed by the provider; we would pay OpenRouter for
  every duplicate even though we refuse to bill the tenant. The
  duplicate must die at the port boundary, before any provider call.

Neither preserves the lens **hexagonal / ports & adapters**.
The current AI surface (`internal/ai/port/ratelimiter.go`,
`internal/ai/port/cache.go`) is already structured this way; the new
LLM port must follow.

## Decision

**A new port `internal/aiassist.LLMClient` defines the single surface
the AI use-cases consume. OpenRouter is the first adapter, lives at
`internal/adapter/llm/openrouter/`, and is the only file in the
repository that imports the OpenRouter SDK. Retry, timeout, and
idempotency are policies enforced by a port-level middleware so every
adapter inherits the same shape.**

### D1 — Port surface

`internal/aiassist/llm_client.go`:

```go
type LLMClient interface {
    Generate(ctx context.Context, req LLMRequest) (LLMResponse, error)
}

type LLMRequest struct {
    TenantID       uuid.UUID
    ConversationID uuid.UUID
    RequestID      uuid.UUID  // generated by caller, stable across retries
    Scope          Scope      // summarise / draft / extract — used by policy + observability
    Prompt         Prompt     // already isolated per [SIN-62225](/SIN/issues/SIN-62225) §3.5
    MaxTokens      int        // upper bound — wallet.Reserve uses this
}

type LLMResponse struct {
    Text         string
    TokensIn     int
    TokensOut    int
    ModelID      string
    LatencyMs    int64
}
```

The use-case in `internal/aiassist/usecase.go` is the only consumer.
It composes:

1. `policy.Resolve(...)` — see [ADR 0042](./0042-policy-cascade.md).
2. `anonymizer.Mask(prompt)` if the resolved policy says so — see
   [ADR 0041](./0041-anonymizer-pii.md).
3. `wallet.Reserve(req.TenantID, req.MaxTokens)` — see [ADR
   0043](./0043-token-debito-atomico.md).
4. `llm.Generate(ctx, req)` — this port.
5. `wallet.Commit(resp.TokensIn + resp.TokensOut)` on success, or
   `wallet.Rollback()` on error.

The use-case never imports OpenRouter, `net/http`, or any vendor SDK.
The lint rules `no-transport-in-domain` and (future)
`no-vendor-sdk-in-domain` enforce this.

### D2 — Retry policy

Three attempts with exponential backoff:

- Attempt 1 immediately.
- Attempt 2 after 250ms (jittered ±20%).
- Attempt 3 after 500ms (jittered ±20%).
- Attempt 4 after 1000ms (jittered ±20%) is **not** taken; total
  retries are capped at 3 so worst-case latency budget stays within
  the 8s p99 below.

Retries fire only on **transport-class** failures and HTTP 5xx /
HTTP 429 from OpenRouter. They do **not** fire on:

- 4xx other than 429 (caller error, e.g. malformed prompt).
- Content-policy block (provider hard error).
- Context-cancellation from the caller (operator closed the panel).

Backoff is implemented by a middleware wrapping the OpenRouter
adapter:

```go
llm := openrouter.New(cfg)
llm = retry.Wrap(llm, retry.Policy{
    MaxAttempts: 3,
    Base:        250 * time.Millisecond,
    Jitter:      0.2,
})
```

The middleware lives at `internal/adapter/llm/retry/` and is portable
across future adapters (Anthropic direct, OpenAI direct, etc.). It is
adapter-side, not domain-side: the use-case sees one `Generate` call
and one error or response.

### D3 — Timeout budget

End-to-end timeout per `Generate` call: **8 seconds at p99**, aligned
with AC #1 of [SIN-62196](/SIN/issues/SIN-62196). The budget is
enforced two ways:

- A `context.WithTimeout(ctx, 8*time.Second)` at the use-case entry
  point. Adapter and middleware must respect `ctx.Done()`.
- A per-attempt deadline of 4 seconds inside the retry middleware so
  the first attempt cannot consume the entire budget and starve
  retries. Math: 4s (attempt 1) + 0.25s sleep + 4s (attempt 2)
  exceeds 8s — therefore attempt 2's per-attempt deadline is
  computed from the remaining budget, not a fixed 4s. The retry
  middleware enforces `min(perAttempt, remainingBudget - jitter)` so
  the third attempt still gets a chance when the second was slow.

If the budget is exhausted, the use-case returns
`aiassist.ErrTimeout` to the caller; the wallet `Rollback` is invoked
(no provider charge persists for the tenant) and an `ai.timeout`
metric is incremented.

### D4 — Idempotency key

The idempotency key is the tuple **`(tenant_id, conversation_id,
request_id)`**.

- `tenant_id` and `conversation_id` are derived from the operator's
  session and the conversation being viewed.
- `request_id` is a UUID generated by the caller (the use-case, on
  first attempt). It is **stable across retries within a single
  `Generate` call** — the retry middleware does not regenerate it.
  It is also stable across a re-issue from the same operator click
  when the front end carries the same `request_id` back to the
  server (Hx-Headers).

The OpenRouter adapter forwards the key in the
`X-Idempotency-Key` HTTP header (OpenRouter's documented contract
for replay-safe POSTs). Format:
`sha256(tenant_id || ':' || conversation_id || ':' || request_id)`
truncated to 32 hex chars. The hash is one-way — the provider
cannot derive tenant identity from the header.

When the provider returns the cached response for a duplicate, the
adapter still increments `ai.idempotent_hit`. The use-case treats
the cached response identically to a fresh one for billing and audit
purposes: the wallet `Commit` sees the same `TokensIn + TokensOut`
the provider reports, which is also the same it reported on the
original successful call.

The caller (the HTTP handler in `internal/http/aiassist/`) is
responsible for generating `request_id` once per operator action and
threading it through the use-case so retries within the same panel
load share a key. The handler stores it in a short-lived
`(tenant_id, conversation_id, operator_id, action_hash) → request_id`
map keyed by the operator's session, with a 60s TTL.

### D5 — Error taxonomy

The port exposes a small typed error set:

- `aiassist.ErrTimeout` — budget exhausted, retry pool drained.
- `aiassist.ErrUnavailable` — adapter cannot reach provider after
  retries (DNS failure, persistent 5xx). Use-case returns this to
  the caller; UI shows a "service unavailable, try again later"
  message and does **not** auto-retry.
- `aiassist.ErrBadRequest` — provider returned a non-retryable 4xx.
  The prompt or model is wrong; the use-case logs the response code
  but **not** the response body (the body may contain echoed
  fragments of an un-anonymised prompt). UI shows a generic
  validation error.
- `aiassist.ErrContentBlocked` — provider's safety filter rejected
  the prompt. UI shows a "this content cannot be processed" message.
- `aiassist.ErrInsufficientBalance` — surfaced by [ADR
  0043](./0043-token-debito-atomico.md), not by the LLM port.

The use-case wraps adapter errors with `errors.Join` to preserve
the underlying cause for observability while exposing the typed
sentinel to callers. `errors.Is`/`errors.As` work as expected.

### D6 — Hexagonal boundary and lints

`internal/aiassist/` contains:

- `llm_client.go` — the port interface, request/response types, and
  sentinel errors.
- `usecase.go` — composition (policy, anonymizer, wallet, llm). No
  transport imports.
- `errors.go` — typed sentinels.

`internal/adapter/llm/openrouter/` contains:

- `client.go` — the one place that imports the OpenRouter SDK and
  constructs `http.Client`.
- `mapping.go` — request/response translation between the port type
  and the provider's wire schema.

`internal/adapter/llm/retry/` contains the per-port retry middleware
(generic over `LLMClient`). It is not domain code; it is an adapter
helper.

Lint rules enforced:

- `no-transport-in-domain`: `internal/aiassist/**` cannot import
  `net/http`, `database/sql`, `github.com/nats-io/...`,
  `github.com/openrouter/...`, or any other transport SDK.
- `no-domain-in-adapter`: adapters may not redefine the port; they
  must satisfy `internal/aiassist.LLMClient`.

## Consequences

Positive:

- **Hexagonal** boundary preserved: the AI use-case sees one
  interface and one error set, regardless of provider. Swapping
  OpenRouter for a self-hosted model is an adapter swap with no
  domain change.
- Retry policy is testable in isolation against a fake `LLMClient`
  that returns scripted transient failures. No live provider needed.
- Idempotency key shape is identical to the wallet's; the same
  `request_id` flows through the entire request and the audit trail
  joins on it. Single key, three layers (provider header, wallet
  ledger, audit log).
- Timeout budget is explicit and bounded. The operator sees a
  failure within 8s in the worst case rather than an open spinner.
- Reversibility: a misbehaving OpenRouter adapter can be replaced or
  wrapped without touching the use-case. If a future provider needs
  a different retry envelope, the middleware is parameterised.

Negative / costs:

- The retry middleware introduces one extra layer to reason about
  when debugging a slow call. Observability (per-attempt span, total
  latency span, retry-count metric) is required and ships with the
  middleware in W2.
- `request_id` must be plumbed through the HTTP handler. A handler
  that forgets and generates a fresh UUID per call breaks
  idempotency; the lint cannot catch this. Mitigation: the handler
  has a thin helper `RequestIDFromAction(...)` and a table-driven
  test asserts the same operator action twice produces the same
  `request_id`.
- An 8s timeout is aggressive for some downstream models. We accept
  the trade-off: any model that cannot respond within 8s should not
  be in front of an operator UI. Background batch flows (Fase 4+)
  will use a different port with a longer budget.

Risk residual:

- A provider that silently changes its idempotency-key semantics
  (e.g. ignores the header, or treats two distinct keys as
  duplicates) would corrupt the audit trail. Mitigation: integration
  test issues two requests with different keys and asserts both
  responses are billed separately; runs nightly against a sandbox
  OpenRouter project.
- A retry storm during a provider outage would amplify load 3×.
  Mitigation: the existing `internal/ai/port/ratelimiter.go`
  (SIN-62225 §3.6-driven, [SIN-62225](/SIN/issues/SIN-62225)) wraps the LLM
  port at the use-case entry point. Rate-limit denials short-circuit
  before the retry loop.

## Alternatives considered

### Option A — Direct SDK call from the use-case

The use-case imports the OpenRouter SDK and hand-rolls retry +
timeout inline.

Rejected because:

- Breaks the hexagonal boundary that the LLM prompt-isolation
  decision ([SIN-62225](/SIN/issues/SIN-62225) §3.4-§3.8)
  established. Domain code would import a vendor SDK.
- Couples retry policy to one provider. Swapping providers means
  rewriting the use-case.
- Untestable without network: every use-case test would need an
  HTTP fake or a full OpenRouter stub.

### Option B — Idempotency at the wallet only

Drop the provider-side header; rely on the wallet's
`(tenant_id, conversation_id, request_id)` UNIQUE constraint to
short-circuit duplicates after the LLM call.

Rejected because:

- The provider would still bill *us* for every duplicate. The wallet
  refusing to bill the tenant means we eat the cost.
- Audit log gains spurious entries for "successful LLM calls that
  were then rolled back," which look like a bug, not a duplicate
  shortcut.
- Misses the cheaper path: OpenRouter's `X-Idempotency-Key` returns
  the cached response without re-invoking the model, saving latency
  and money.

### Option C — Synchronous provider call, no retry

Make the use-case fail on the first transport error and rely on the
operator to retry by clicking the button again.

Rejected because:

- 2–5% transient-failure rate against OpenRouter in our smoke tests.
  Every transient blips through to the operator as a hard failure.
- The operator clicking again generates a *new* `request_id`
  (handler regenerates by default), which would re-bill on retry.
  We would have to push the dedup window into the handler — which
  is exactly what D4 does, but cleaner because retries are inside
  the same `request_id`.

## Lenses cited

- **Hexagonal / ports & adapters.** Single port surface; one adapter
  per provider; retry and timeout are middleware, not domain.
- **Idiomatic Go.** Sentinel errors expose `errors.Is` matching,
  context propagation throughout, small interface (one method),
  retry middleware is a closure over the wrapped port.
- **Reversibility & blast radius.** A provider swap or retry-policy
  change is a one-file adapter edit. A bug in retry does not affect
  the wallet or the audit pipeline.
- **Boring technology budget.** `context.WithTimeout`, exponential
  backoff with jitter, an HTTP header for idempotency. Nothing
  novel; everything in the Go stdlib or a small helper package.
- **Observability before optimisation.** Per-attempt span, retry
  count, idempotent-hit count, and timeout count are required
  metrics; they ship with the middleware.

## Out of scope

- Streaming responses. Fase 3 returns a single complete response per
  call. Streaming is a separate ADR when the UI grows a token-by-
  token surface.
- Multi-provider fallback (OpenRouter → Anthropic direct on
  outage). The port supports it conceptually but Fase 3 ships one
  adapter only.
- Long-running background jobs (Fase 4+ batch summarisation). They
  will use a different port with a different timeout envelope.
